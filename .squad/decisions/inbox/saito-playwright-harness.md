# Decision: Playwright Harness for ONNX Inference Performance Tests

**Author:** Saito  
**Date:** 2026-03-07  
**Issue:** #23  
**PR:** #24  

## Decision

Use Playwright E2E (not Playwright-CT) with Vite dev server as webServer. Tests inject the `ort.wasm.min.js` UMD bundle via `page.addScriptTag()` and intercept all model + WASM file requests via `page.route()` to serve a minimal stub ONNX model from `tests/fixtures/`.

## Rationale

- **No app modification required**: Test runs ONNX inference independently via injected ort bundle, without needing to modify App.tsx or expose internal state.
- **Vite dev server**: Serves TypeScript source directly, avoids build step in the test loop. `reuseExistingServer: !CI` means local re-runs are fast.
- **Stub model isolation**: 345-byte stub model (Slice + Reshape ops) means load/inference times reflect pure WASM overhead — no network latency, no large file downloads in CI.
- **WASM file interception**: `page.route('**/ort-wasm-dist/**')` intercepts WASM requests and serves them from `node_modules/onnxruntime-web/dist/` — no CDN required, works fully offline in CI.
- **Single-thread mode**: `ort.env.wasm.numThreads = 1` avoids SharedArrayBuffer COOP/COEP headers required for multi-threaded WASM in CI headless mode.

## Alternatives Rejected

- **Playwright-CT (`@playwright/experimental-ct-react`)**: Would require a separate config and test fixtures per component. More complexity for what is essentially a performance harness, not a component unit test.
- **`vite preview` (built files)**: Would require a build step before tests. Vite dev server is faster for the dev loop. Playwright's `webServer` option handles startup automatically.
- **Real model in CI**: 53 MB ONNX file would require LFS, slow CI, and fail without a backend. Stub model is the right CI approach.

## Stub Model Spec

- File: `tests/fixtures/stub-generator.onnx` (345 bytes)
- Generated by: `python -c "import onnx; ..."` (see PR description for full script)
- Inputs: `style_glyphs [1,10,1,128,128] float32`, `char_index [1] int64`
- Output: `generated_glyph [1,1,128,128] float32`
- Ops: `Slice` (dim 1, indices 0→1) + `Reshape` to [1,1,128,128]
- Output range: always equal to first style glyph slice value (valid float32 in [-1,1] when input is in [-1,1])

## Test Results

- **Chromium**: 17/17 pass, 5.1s total
- **Firefox**: 17/17 pass, 18s total (WASM JIT compile overhead expected)
- **WebKit/Safari**: 17/17 pass, 1.7s total

## Performance Observed (stub model, CI)

- InferenceSession.create (WASM): ~300ms (Chromium), ~800ms (Firefox)
- Per-glyph inference: ~10ms (Chromium), ~60ms (Firefox) — well within 500ms target
- 66-glyph total: ~400ms (Chromium), ~2500ms (Firefox) — within 10s target
